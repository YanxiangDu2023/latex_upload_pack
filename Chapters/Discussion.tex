\section{Answer to the Research Questions}







\subsection{Answer to Sub-RQ1: How can the architecture and data handling of a web-based prototype be designed to support structured upload, storage, and management of heterogeneous workflow inputs, including source datasets and matrix-based business rules?}

Sub-RQ1 was answered by turning file handling into an explicit part of the prototype rather than treating it as a separate preparatory task. In the implemented system, source datasets and matrix files have dedicated upload and management views, and each file is validated, normalized, and stored together with metadata such as type, timestamp, row count, processing status, and version information. This matters in the Volvo CE case because later stages depend on knowing exactly which source file or matrix version was used.

The practical effect is that users can do more than just upload a file once and move on. They can inspect rows in the interface, download stored files, save edited content as a new version, and check whether the required inputs are present before running downstream stages. That was an important design choice because unclear file state is itself part of the traceability problem in workflows built around recurring yearly runs and maintained business rules.

In the 2024 evaluation, the prototype handled different source and matrix files correctly across upload, storage, retrieval, download, and re-save operations. Within the scope of the Volvo CE case, this shows that heterogeneous inputs can be managed in a more controlled and inspectable way than ad hoc file exchange, while also pointing to a more general design approach for input management in ERP-based enterprise workflows.

\subsection{Answer to Sub-RQ2: How can staged data-processing activities be re-expressed in a web-based environment so that intermediate transformations and derived results become inspectable and traceable for business users?}

Sub-RQ2 was answered by moving selected process stages out of a largely implicit SAP representation and into explicit web-based stages with persisted intermediate outputs. In the Volvo CE case, this meant implementing preparation, market view, adjustment, machine line split, and resplit as named stages that users can run and inspect separately.

What changed is not only where the logic runs, but how it can be reviewed. In the preparation stage, users can see how CRP, SAL, and OTH records are aligned with mappings and control fields before later calculations continue. In market view, they can inspect how prepared data is reorganized into the market-oriented structure used for later steps. In adjustment, source-detail and derived rows are shown together instead of being hidden behind a final result. In machine line split and resplit, the prototype exposes split ratios, before-and-after totals, and redistributed rows, which makes it easier to understand why one grouped value was broken into several output rows.

The inspection views were a necessary part of this answer. Pipeline Viewer shows stage order, readiness, and dependencies, while Layer Detail pages allow row-level inspection, summaries, and review of rule-related fields. In the Volvo CE case, this made it possible to discuss concrete intermediate results rather than only final outputs. More broadly, it shows one way of re-expressing ERP-based workflow logic so that business users can inspect the transformation path instead of treating it as a black box.

\subsection{Answer to Sub-RQ3: To what extent does the implemented prototype improve transparency, usability, and traceability compared to an existing SAP-based TMC implementation, as assessed through demonstration and stakeholder feedback?}

Sub-RQ3 was answered positively within the implemented scope, but the answer depends on what was actually evaluated. The prototype was not compared with SAP as a full enterprise platform. Instead, it was compared with the existing way in which the selected TMC stages are inspected and discussed from a business-user perspective.

In the SAP-based TMC implementation, the process is carried out through technical calculation steps and system objects such as matrices, planning functions, planning sequences, and reporting objects. Those structures can execute the process, but they do not give business users a simple way to inspect how CRP, SAL, OTH, and matrix-based rules shaped an intermediate or final output.

The prototype improved that situation by putting the selected stages into one inspection-oriented environment. Upload and matrix-management pages made input handling easier to review, Pipeline Viewer made stage dependencies visible, and Layer Detail pages exposed row-level outputs, summaries, and rule-related fields. During demonstration with 2024 case data, this made it easier to check whether a result matched the expected logic and to locate where a mismatch appeared when it did not.

Stakeholder feedback pointed in the same direction. The separated stage views made the process easier to explain and easier to check than the existing SAP-based inspection situation for the same scope. The evaluation therefore supports the claim that the prototype improved transparency, usability, and traceability for the implemented stages, while still leaving open how far the same improvement would carry to a broader production setting.

More broadly, the Volvo CE case suggests that inspection-oriented representations can add value in ERP-based enterprise workflows when business users need to see how inputs, maintained rules, and staged transformations connect to the reported result.

\section{Contributions and Originality}
\subsection{Practical Contributions and Originality}

The main practical contribution of this thesis is the development of a working prototype that lets users inspect selected TMC stages in a way that was not previously available in the Volvo CE setting. Rather than replacing SAP, the prototype complements the existing implementation with an environment for upload review, staged inspection, and discussion of intermediate results.

Its originality lies in how the selected stages were re-expressed for inspection. Preparation, market view, adjustment, machine line split, and resplit are shown as explicit reviewable stages, and users can work with filterable tables, summaries, charts, and CSV export instead of relying only on final aggregated outputs or technical SAP structures. The contribution is therefore not a new algorithm. It is a concrete design contribution that shows one way to make an enterprise workflow more inspectable when it depends on heterogeneous inputs, maintained rules, and sequential transformations.

\subsection{Academic Contributions}
On the academic side, the thesis contributes a concrete example of how transparency, traceability, usability, and inspectability problems in enterprise data-processing workflows can be addressed through an implemented artefact rather than only discussed conceptually. The Volvo CE case is specific, but the underlying issue is wider: many enterprise workflows depend on heterogeneous inputs, staged transformations, and embedded rules that are difficult for business users to inspect \cite{foidl2024pipeline,putrama2024heterogeneous,vassiliadis2009etl,weber2022sap,calisir2004erp}.

The academic contribution is therefore best understood as design knowledge, not as a new algorithm. In this case, the study shows that structured input management, explicit intermediate outputs, and stage-oriented inspection views can support validation, explanation, and error investigation in workflows that would otherwise be difficult to follow from a business-user perspective. The transferable value lies in that design approach, even though the present evidence still comes from one case.

Methodologically, the thesis also shows how a DSR project can stay grounded in a real organizational workflow while still moving beyond pure description. The Volvo CE case was used to identify the problem, clarify requirements, guide implementation choices, and evaluate the resulting artefact \cite{johannesson2021}. In that sense, the case was not only a background story; it shaped the design work throughout the study.
\section{Limitations}

Although the artefact was developed and demonstrated as a working prototype, several limitations remain.

First, the evaluation was conducted in a controlled prototype setting using selected 2024 TMC process data. This means that the study should be understood as a single-case evaluation rather than as universal proof that the same design will work equally well across all enterprise contexts. The system has not yet been tested as a regular internal tool with broader yearly data, different workflow variations, or day-to-day business use.

Second, the implemented scope covers only selected parts of the case workflow, including preparation, market view, adjustment, machine line split, and resplit. Full Total Market Calculation, full Restatement, and final Reporting were not implemented as production-level stages. The prototype should therefore be understood as a partial workflow inspection and validation tool rather than a full replacement of the SAP-based process.

Third, some rule changes are still dependent on backend development. Certain mappings, workflow conditions, and calculation logic are implemented in SQL queries and Python code, so business users cannot yet configure all rules directly through the interface.

Fourth, the prototype supports stage-level traceability, but not full end-to-end row-level lineage. Key context fields are preserved in the outputs, but the system does not yet provide a complete audit trail from every output row back to the original input rows and rule applications.

Finally, the prototype has not yet been evaluated for production-level deployment or large-scale multi-user use. Broader internal use would require authentication, monitoring, backup and recovery, process orchestration, and deployment within Volvo CE's internal infrastructure.


\section{Ethical and Societal Implications}

The artefact provides a more accessible way to process and inspect selected parts of the Volvo CE TMC process. The main ethical considerations concern responsible handling of company-internal data and avoiding overreliance on generated outputs. Since the results still depend on uploaded data, mappings, and implemented rules, the artefact should be treated as a validation and inspection tool rather than as a guarantee of correctness \cite{goddard2012automation}. Future use should therefore keep intermediate steps and assumptions visible so that errors can be traced and reviewed rather than hidden behind final outputs \cite{martin2019ethics}.



\section{Software Tools and the Use of AI Tools}

Several software tools were used during the development of the artefact and the writing of the thesis.

Codex was used as a supportive tool during development, mainly for exploring implementation alternatives, discussing frontend-backend interactions, and assisting with SQL, API logic, and debugging. While it helped improve efficiency, all system design decisions and final implementations were made by the author.

ChatGPT was used during the writing process to refine language, improve clarity, and adjust phrasing to a more academic style. As English is not the author's first language, it was particularly useful for improving grammar and sentence structure. However, all ideas and conclusions were developed independently.

Visual Studio Code served as the main development environment, draw.io was used for diagrams, and Overleaf was used for writing and formatting the thesis in \LaTeX. These tools supported the development and documentation process but did not influence the research direction or results.





\section{Future Work}

Future work should address both system scope and research scope. On the system side, the artefact could be extended to later workflow stages such as full Total Market Calculation, Restatement, and Reporting, so that a larger part of the workflow can be inspected in one environment. Another important step would be to move more mappings, workflow conditions, and calculation rules out of backend code and into configurable structures, which would make the system easier to maintain as rules change over time.

Further work is also needed on traceability. The current prototype exposes intermediate stages, but it does not yet provide full end-to-end row-level lineage. A useful continuation would therefore be to strengthen drill-down from final outputs to original inputs, make rule applications more explicit, and support comparison between workflow runs. This would improve the artefact not only as a review tool, but also as a support tool for investigation and explanation.

Finally, the artefact should be studied over a longer period in regular organizational use. This would make it possible to assess its effects on validation work, communication between business and IT, and user confidence in workflow outputs. It would also allow further examination of whether the same inspection-oriented design approach can be transferred to other enterprise data-processing workflows beyond the Volvo CE case. A further research step would therefore be to apply the same design approach in another organizational setting in order to distinguish more clearly between case-specific findings and more transferable design principles.
