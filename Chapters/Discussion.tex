\subsection{Answer to Sub-RQ1: How can the architecture and data handling of a web-based prototype be designed to support structured upload, storage, and management of heterogeneous TMC workflow inputs, including source datasets and matrix-based business rules?}


Sub-RQ1 is addressed by implementing a web-based input management layer that supports structured upload, review, and versioning of workflow files. The prototype provides dedicated upload interfaces for both source datasets and matrix inputs. Uploaded files are validated, normalised, and stored together with metadata such as file type, timestamp, processing status, row count, and version information. This makes it possible to track which input files were used and to retrieve the latest valid version of each required dataset.

The frontend also supports practical management actions such as viewing uploaded data, downloading stored inputs, and saving edited data as a new version. In this way, input handling becomes a traceable process rather than a one-time file upload. Users can inspect rows directly in the interface and verify whether required inputs are available before running downstream workflow steps.

The 2024 evaluation showed that different source and matrix files could be ingested, stored, retrieved, downloaded, and re-saved correctly. Therefore, the artefact answers Sub-RQ1 by making workflow inputs easier to access, manage, inspect, and reuse through a structured web-based interface.

\subsection{Sub-RQ2: How can the preparation, adjustment, and machine line split stages of the TMC workflow be re-expressed in a web-based environment so that intermediate transformations and derived results become inspectable
and traceable for business users?}

Sub-RQ2 asked how the preparation, adjustment, and machine line split stages of the TMC workflow can be re-expressed in a web-based environment so that intermediate transformations and derived results become inspectable and traceable for business users.

The prototype addresses this by representing the selected TMC stages as explicit web-based workflow layers with persisted intermediate outputs and stage-specific inspection views.

\begin{itemize}
    \item \textbf{Preparation:} The preparation layer validates, cleans, and aligns CRP, SAL, and OTH data with matrix-based rules, including source, country grouping, machine line, brand, and size class mappings. It exposes mapped dimensions and control fields such as reporter and deletion-related flags, making raw-row interpretation inspectable before downstream processing.

    \item \textbf{Adjustment:} The adjustment layer represents regrouping and derived-value generation as a separate reviewable step. It provides both source-detail and derived rows in a common output structure, supporting traceability at row and aggregate levels.

    \item \textbf{Machine line split:} The split layer makes redistribution logic explicit by transforming grouped OTH values into finer machine-line and size-class categories using rule-based split ratios. Before/after values and split details are exposed so users can verify the redistribution process.

    \item \textbf{Inspection mechanism:} Pipeline Viewer provides stage status, readiness, and dependency information, while Layer Detail pages provide row-level outputs, summaries, and rule-related fields. Together, these views connect workflow-level monitoring with record-level validation.
\end{itemize}

Overall, the layered web representation turns selected SAP-embedded transformations into explicit, inspectable, and traceable workflow stages.

\subsection{Answer to Sub-RQ3: To what extent does the implemented prototype improve transparency, usability, and traceability in the TMC workflow compared to the existing SAP-based representation, as assessed through demonstration and stakeholder feedback?}

Sub-RQ3 asked to what extent the implemented prototype improves transparency, usability, and traceability in the TMC workflow compared with the existing SAP-based representation.

In the SAP-based representation, the TMC workflow is organised through technical calculation steps and system objects such as matrix structures, planning functions, planning sequences, and reporting objects. Although the process can be executed in SAP, business-side inspection of intermediate logic is less direct, especially when users need to trace how outputs are formed from CRP, SAL, OTH, and matrix-based rules.

The prototype improves this representation by exposing the workflow through dedicated web-based inspection flows. Input handling is centralised through upload and matrix-management pages, workflow progress is shown through Pipeline Viewer, and intermediate results are made available through Layer Detail pages with row-level outputs, summaries, and rule-related fields. This makes preparation, adjustment, and split-related transformations easier to inspect, validate, and discuss.

The improvement was assessed through demonstration with 2024 workflow data and stakeholder feedback. Within the evaluated scope, the prototype made intermediate stages more visible, reduced ambiguity in result checking, and supported more practical validation of derived outputs. Stakeholder feedback indicated that the separated workflow views helped users follow dependencies, inspect intermediate results, and discuss rule behaviour more directly than in the SAP-based workflow.

\section{Contributions and Originality}
\subsection{Practical Contributions}

The main practical contribution of this thesis is the development of a working web-based prototype for selected stages of the Volvo CE Total Market Calculation (TMC) workflow. The artefact provides a more transparent and inspectable complement to the existing SAP-based execution and review process. Rather than replacing SAP, the prototype demonstrates how selected parts of an SAP-based enterprise workflow can be re-expressed in a user-oriented web environment to support validation, inspection, and communication.

A key contribution is that the prototype exposes intermediate workflow outputs, including preparation, adjustment, and split-related results, instead of only presenting final aggregated figures. Through filterable tables, summary values, charts, and CSV export, users can inspect how values are formed, compare outputs with source data, and validate whether the implemented logic behaves as expected.

For the Strategy and Business Intelligence function at Volvo CE, the prototype offers a new digital way of working with TMC-related data. It supports process transparency, independent review, and more efficient discussion of workflow logic and data-quality issues. In this sense, the artefact provides practical support for business users who need to understand and validate complex enterprise data transformations without relying only on technical system interpretation.

\subsection{Originality}

The originality of this thesis lies in the independent design and development of a dedicated web-based prototype for the studied Volvo CE TMC context. Before this work, there was no specific solution in this context that exposed selected TMC workflow stages as inspectable processing layers outside the existing SAP-based environment.

The prototype is original in its practical re-expression of preparation, adjustment, and split-related logic into a transparent web-based inspection environment. Instead of only reviewing final aggregated outputs, users can inspect intermediate results through tables, summaries, charts, and exportable data.

The contribution is not a new algorithm, but a context-specific solution that demonstrates how embedded SAP-based workflow logic can be complemented by a more transparent, traceable, and usable prototype.

\section{Limitations}

Although the artefact was developed and demonstrated as a working prototype, several limitations remain.

First, the evaluation was conducted in a controlled prototype setting using selected 2024 TMC workflow data. The system has not yet been tested as a regular internal tool with broader yearly data, different workflow variations, or day-to-day business use.

Second, the implemented scope only covers selected parts of the TMC workflow, including preparation, market-view derivation, adjustment, and selected split/resplit-related processing. Full Total Market Calculation, full Restatement, and final Reporting were not implemented as complete production-level stages. Therefore, the prototype should be understood as a partial workflow inspection and validation tool rather than a full replacement of the existing SAP-based process.

Third, some rule changes are still dependent on backend development. Certain mappings, workflow conditions, and calculation logic are implemented in SQL queries and Python code, so business users cannot yet configure all rules directly through the interface.

Fourth, the prototype supports stage-level traceability, but not full end-to-end row-level lineage. Key context fields such as country, machine line, source, brand, flags, split level, and split ratio are preserved in the outputs. However, the system does not yet provide a complete audit trail from every output row back to the original input rows and rule applications.

Finally, the prototype has not yet been evaluated for production-level deployment or large-scale multi-user use. Although the system can be configured with PostgreSQL, broader internal use would require internal authentication, monitoring, backup and recovery, process orchestration, and deployment within Volvo CE's internal infrastructure.


\section{Ethical and Societal Implications}

The artefact provides a more accessible way to process and inspect selected parts of the Volvo CE TMC workflow. By exposing inputs, intermediate steps, rule-related fields, and generated outputs, it can support transparency and make validation easier for business users.

The main ethical consideration concerns company-internal data and business logic. Since the TMC workflow includes market-related data, internal mappings, and rule structures specific to Volvo CE, the prototype was developed and demonstrated in a controlled setting, and sensitive details were handled carefully in the thesis material.

Another consideration is overreliance on generated outputs. The results produced by the artefact still depend on the quality of uploaded data, mappings, and implemented rules. Therefore, the artefact should be treated as a validation and inspection tool, rather than as a system that automatically guarantees correctness \cite{goddard2012automation}.

Finally, responsibility for incorrect results may involve input data, mapping files, rule interpretation, backend implementation, or user interpretation. Future use should therefore keep intermediate steps and assumptions visible, so that errors can be traced and reviewed rather than hidden behind final outputs \cite{martin2019ethics}.


\section{Future Work}

Several directions for future work follow from the current prototype. One important direction is to extend the artefact beyond the currently implemented stages. Future versions could cover full Total Market Calculation, full Restatement, and final Reporting, so that the prototype can develop from a partial workflow inspection tool into a more complete end-to-end representation of the TMC process.

Rule handling is another area for improvement. In the current implementation, some mappings, workflow conditions, and selected calculation rules are still partly embedded in backend SQL queries and Python processing logic. A future version could move more of these rules into configurable data structures or administrative interfaces. This would make selected workflow logic easier to adjust without requiring direct code changes.

The interaction model could also be developed further. Although the current frontend supports filtering, dynamic summaries, chart-based views, and CSV export, future versions could add richer drill-down functionality, cross-view linking, configurable comparison modes, and more detailed step-by-step inspection of intermediate results and derived metrics. These functions would strengthen the artefact's role as a validation and investigation tool for business users.

Finally, deployment, scalability, and enterprise integration would need to be strengthened before broader internal use. The current prototype supports a relational database setup and can be configured for PostgreSQL, but it has not yet been evaluated as a production-level multi-user system within Volvo CE's internal infrastructure. Future work should therefore address service architecture, container-based deployment, environment configuration, backend process orchestration, monitoring and logging, backup and recovery, and integration with internal authentication, authorization, and hosting infrastructure.
